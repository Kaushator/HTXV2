# CI/CD процесс для проекта HTXV2

В этом документе описан процесс непрерывной интеграции и доставки (CI/CD), настроенный для проекта HTXV2.

## Обзор CI/CD процесса

CI/CD процесс проекта HTXV2 состоит из следующих компонентов:

1. **Автоматические проверки кода** при создании Pull Request или коммите в основные ветки
2. **Автоматическая оптимизация кода** с помощью линтеров и форматеров
3. **Автоматическое тестирование** бэкенда и фронтенда
4. **Проверки безопасности** с помощью Trivy
5. **Отчеты о состоянии проекта** по расписанию
6. **Автоматическое обновление зависимостей** с помощью Dependabot

## Настроенные рабочие процессы

### 1. Проверка бэкенда

**Файл**: `.github/workflows/backend-checks.yml`

**Запускается**:

- При push в ветки `main`, `master`, `dev`, затрагивающем файлы в директории `backend/`
- При создании PR в указанные ветки, затрагивающем файлы в директории `backend/`

**Действия**:

- Установка Python и зависимостей
- Проверка кода с помощью ruff
- Проверка форматирования с помощью black
- Запуск тестов с помощью pytest

### 2. Проверка фронтенда

**Файл**: `.github/workflows/frontend-checks.yml`

**Запускается**:

- При push в ветки `main`, `master`, `dev`, затрагивающем файлы в директории `frontend/`
- При создании PR в указанные ветки, затрагивающем файлы в директории `frontend/`

**Действия**:

- Установка Node.js и зависимостей
- Проверка кода с помощью ESLint
- Запуск тестов

### 3. Оптимизация кода

**Файл**: `.github/workflows/code-quality.yml`

**Запускается**:

- При push в ветки `main`, `master`, `dev`
- При создании PR в указанные ветки
- Вручную через интерфейс GitHub (workflow_dispatch)

**Действия**:

- **Бэкенд**:
  - Форматирование кода с помощью black
  - Сортировка импортов с помощью isort
  - Автоматические исправления с помощью ruff
  - Автоматический коммит исправлений
- **Фронтенд**:
  - Автоматические исправления с помощью ESLint
  - Форматирование кода с помощью Prettier
  - Автоматический коммит исправлений

### 4. Проверка безопасности

**Часть workflow**: `.github/workflows/code-quality.yml`

**Действия**:

- Сканирование кода на уязвимости с помощью Trivy
- Загрузка результатов в формате SARIF

### 5. Отчеты о состоянии проекта

**Файл**: `.github/workflows/status-report.yml`

**Запускается**:

- По расписанию (ежедневно в полночь по UTC)
- Вручную через интерфейс GitHub (workflow_dispatch)

**Действия**:

- Генерация отчета о состоянии проекта
- Отправка отчета по email указанным получателям

## Автоматическое обновление зависимостей

Для автоматического обновления зависимостей настроен Dependabot в файле `.github/dependabot.yml`:

- **Бэкенд**: Еженедельная проверка зависимостей Python
- **Фронтенд**: Еженедельная проверка зависимостей npm
- **GitHub Actions**: Ежемесячная проверка обновлений для Actions

## Локальное тестирование Actions

Для локального тестирования GitHub Actions используется инструмент [act](https://github.com/nektos/act).
В проекте добавлены удобные скрипты для запуска тестирования:

**PowerShell**:

```powershell
.\.devcontainer\scripts\powershell\github\github-actions-test.ps1
```

**Bash**:

```bash
./.devcontainer/scripts/github/github-actions-test.sh
```

## Оптимизация кода локально

Для запуска оптимизации кода локально используйте:

**PowerShell**:

```powershell
.\.devcontainer\scripts\powershell\github\optimize-code.ps1 -Target All
```

**Bash**:

```bash
./.devcontainer/scripts/github/optimize-code.sh --all
```

## Автоматическое создание PR

Для автоматического создания PR с AI-генерированным описанием используйте:

**PowerShell**:

```powershell
.\.devcontainer\scripts\powershell\github\auto-pr.ps1 -Title "Название PR" -Branch "feature/my-feature"
```

**Bash**:

```bash
./.devcontainer/scripts/github/auto-pr.sh --title "Название PR" --branch "feature/my-feature"
```

## Лучшие практики работы с CI/CD

1. **Часто коммитьте и пушьте изменения** в свои feature-ветки для раннего выявления проблем.
2. **Проверяйте результаты автоматических проверок** в Pull Request перед мерджем.
3. **Запускайте оптимизацию кода локально** перед отправкой изменений, чтобы избежать лишних коммитов от автоматических исправлений.
4. **Обращайте внимание на отчеты безопасности** и своевременно устраняйте уязвимости.
5. **Принимайте PR от Dependabot** после убеждения, что они не ломают функциональность.

## Настройка новых GitHub Actions

Если требуется добавить новый рабочий процесс:

1. Создайте новый файл в директории `.github/workflows/` с расширением `.yml`.
2. Следуйте структуре существующих файлов для определения событий запуска и шагов.
3. Протестируйте локально с помощью `act` перед отправкой на GitHub.
4. Обновите этот документ, добавив информацию о новом рабочем процессе.

## Настройка секретов

Для корректной работы некоторых рабочих процессов требуется настроить секреты в настройках репозитория на GitHub:

1. **Отправка email-отчетов**:
   - `MAIL_SERVER`: SMTP-сервер
   - `MAIL_PORT`: Порт SMTP-сервера
   - `MAIL_USERNAME`: Имя пользователя для авторизации на SMTP-сервере
   - `MAIL_PASSWORD`: Пароль для авторизации
   - `MAIL_RECIPIENTS`: Список получателей отчетов (через запятую)

## Решение проблем

### Ошибки в GitHub Actions

1. Проверьте логи выполнения workflow в интерфейсе GitHub
2. Попробуйте воспроизвести проблему локально с помощью `act`
3. Проверьте наличие всех необходимых секретов и переменных

### Проблемы с локальным запуском act

1. Убедитесь, что Docker запущен
2. Проверьте версию act (рекомендуется последняя)
3. Проверьте наличие необходимых переменных окружения
